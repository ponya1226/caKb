# ADR 0005: 家族共有向けクラウド正本化方針

Date: 2026-07-05

## Status

Accepted

## Context

caKbは個人利用向けのローカル保存PWAとして開始した。Google Vision OCRの導入により外部OCR Providerは任意利用できるようになったが、支出データの正本は引き続きブラウザ内IndexedDBにある。

今後は家族からもレシート登録できるようにし、認証、家族共有、既存スプレッドシートとの同期、自由なカテゴリ作成、年間支出一覧を扱いたい。これらは複数端末・複数利用者で同じ家計簿を共有するため、IndexedDBだけを正本にすると同期衝突や端末消失時の復旧が難しい。

## Decision

次フェーズでは、家計簿データの正本をクラウドへ移す方針とする。第一候補はFirebase AuthとCloud Firestoreを使う。

- 認証はFirebase AuthのGoogleログインを第一候補にする。
- 支出、カテゴリ、店舗別カテゴリルール、家計簿メンバー情報はCloud Firestoreを正本にする。
- IndexedDBは未ログイン時のローカル利用、移行元、将来のオフラインキャッシュとして扱う。
- 既存ローカルデータは初回ログイン後にユーザー操作でクラウドへ移行する。自動移行はしない。
- スプレッドシート連携は、まずアプリ正本からGoogle Sheetsへ出力する一方向同期とする。
- 双方向同期、家族権限の細分化、レシート画像のクラウド保存は別判断とする。
- 家族参加は、Googleログイン後に管理者が発行した期限付き招待コードを入力する方式とする。
- 招待コードは24時間有効、1回限りとし、参加時に招待消費、member作成、active household更新を同一transactionで行う。
- 初期権限は `owner` と `member` の2種類とし、管理者だけが招待発行とメンバー解除を行える。

## Consequences

- MVPの「ローカル保存のみ」方針から、次フェーズでは「認証あり・クラウド正本あり」へ移行する。
- Firestore Security Rules、認証状態、家計簿メンバー権限の設計が必要になる。
- Google Vision ProxyはFirebase ID token検証などの利用制限を追加する必要がある。
- Google Sheets同期では、Sheets側編集の取り込みを行わないため、初期版は衝突処理を単純にできる。
- Firebase、Firestore、Sheets APIの無料枠を前提に始めるが、利用量によって課金が発生する可能性がある。
- 招待コード文書はコードを知るログインユーザーによる単一取得だけを許可し、一覧取得はSecurity Rulesで拒否する。

## Alternatives

- IndexedDBを正本にして端末間同期だけ追加する: ローカル方針は保てるが、家族共有と衝突解決が複雑になる。
- 既存スプレッドシートを正本にする: 移行は直感的だが、アプリ側の検索、集計、オフライン利用、認証制御が弱くなる。
- Supabase Auth / Databaseを使う: 認証とDBをまとめられるが、既存のGoogle Cloud Vision / Sheets連携との基盤統一ではFirebaseの方が自然。
- 自前認証を実装する: 自由度は高いが、セキュリティと運用負荷が大きいため採用しない。
